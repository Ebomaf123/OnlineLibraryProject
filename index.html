Advanced Programming Project: Online Library

Implementation of a complete software solution for managing multimedia metadata (books, movies, magazines) via a RESTful API and a dedicated Graphical User Interface (GUI).

View Technical Architecture

1. Project Context and Objectives (Task 1)

The objective is to design a robust system enabling the creation, retrieval, and management of multimedia metadata. This involves applying advanced programming concepts, particularly inter-module communication (Frontend and Backend) via HTTP requests.

Managed Metadata

Name, Publication Date, Author, Category.

Media Categories

Book, Movie, Magazine.

Data Persistence

Data permanently stored in a JSON or Pickle file.

2. Architecture and Communication (Task 2)

Backend: Flask (Python)

The Flask backend manages data storage and persistence (JSON/Pickle), and exposes a RESTful API. It handles all CRUD operations (Create, Read, Update, Delete) required by the system.

Example API Endpoints:
  • GET /api/media List all media items
  • GET /api/media/category/movie Filter by category
  • POST /api/media Create a new item
  • DELETE /api/media/{id} Delete an item

Frontend: GUI (PyQt/PySide/Tkinter)

The graphical interface is the client-side application. It initiates HTTP requests to the Flask backend for all data operations and displays the results in an organized interface.

Interface Features:
  • Display of the complete list and filtered views.
  • Search field by exact name.
  • Detailed view of metadata by clicking an item.
  • Form for creating new media records.

3. User Interface (GUI) Overview

Main View and Filtering

The main screen presents a searchable and filterable list. Users can easily browse media and apply filters by category (Book, Movie, Magazine) or via a text search.

  • Display of API results in a clear list.
  • Management of user interactivity (click, search, filter).

Media Creation and Detail View

This view presents the form for creating a new item. The form handles data validation (e.g., date format) before sending the metadata to the backend via a POST or PUT (update) request.

  • Input validation to ensure data integrity (e.g., date format).
  • Clear separation between presentation logic (GUI) and business logic (API).

4. Testing, Error Handling, and Best Practices (Tasks 3 & 4)

Quality assurance is a pillar of the project. The solution integrates unit and integration tests, as well as exception handling mechanisms to ensure its robustness and maintainability.

Unit and Integration Tests (Task 3)

At least three tests are implemented to verify core functionality, covering both backend and frontend:

  • Backend Test: Verification of the POST /api/media endpoint for correct data serialization and persistence.
  • Frontend Test: Simulation of a user search to ensure the correct result is displayed based on the API response.
  • Error Test: Verification of handling invalid input (e.g., attempting to create without a required field).

Exception Handling (Task 4)

Exception handling and best practices are applied at all levels:

  • API Layer: Handling 404 (not found) codes for element searches and 400 (bad request) for invalid JSON data.
  • GUI Layer: Displaying clear error messages to the user (via a modal box) instead of crashing, especially for network connection errors.
  • Code Quality: Use of meaningful names, clear modular structure, and detailed comments, as requested in the assignment.
